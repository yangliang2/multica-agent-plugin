## 1. Context

v0.1.0 assumed a single-agent issue. v0.2.0 introduces Squad-aware behavior: when multica daemon claims an issue assigned to a squad, the leader's agent `Instructions` field has already been authored by the workspace operator (via `multica squad update --instructions`) and contains the literal heading `## Squad Operating Protocol` plus a roster block. The daemon then renders an **on-disk** runtime config file (CLAUDE.md for Claude Code) that embeds those instructions verbatim under `## Agent Identity`. The plugin's job is to detect this on-disk marker and switch into a leader workflow (coordinate, do not execute) while still supporting member and single-agent paths transparently.

**Source-truth-verified facts (re-confirmed against multica HEAD on 2026-05-23):**

- Daemon-side leader detection: `IsSquadLeader = strings.Contains(instructions, "## Squad Operating Protocol")` at `server/internal/daemon/daemon.go:2260`. The same check exists at `server/internal/daemon/prompt.go:145`. Two source-of-truth check-sites in the daemon mean the marker text is genuinely the contract.
- The daemon writes the agent's effective config to disk **every time a task is claimed** (Claude provider): `server/internal/daemon/execenv/runtime_config.go:101-106` — `InjectRuntimeConfig(workDir, "claude", ctx)` → `os.WriteFile(filepath.Join(workDir, "CLAUDE.md"), ...)`. The body is built by `buildMetaSkillContent` (`runtime_config.go:117`) and at line 137 it emits `ctx.AgentInstructions` verbatim. Therefore `## Squad Operating Protocol`, if present in `instructions`, **WILL appear in `{workDir}/CLAUDE.md`** for every Claude leader claim.
- `multica squad activity <issue-id> <action|no_action|failed> --reason "..."` is implemented at `server/cmd/multica/cmd_squad.go:379-434`. On success it prints `Squad evaluation recorded: <outcome> (issue <display>)` to **stderr** (line 427). Valid outcomes are enforced by the CLI at line 399.
- `multica issue create` accepts both `--status` and `--assignee-id` flags (`server/cmd/multica/cmd_issue.go:274, 277`). Strategy A (child issue + assignee) is therefore CLI-supported.
- Briefing is also injected to the underlying agent process via `--append-system-prompt` (claude.AgentInstructions); but the plugin's **only reliable read surface** is the on-disk CLAUDE.md, not the in-memory system prompt.
- Concurrency: per-agent `max_concurrent_tasks=6` cap; (agent,issue) is serial; different agents fully parallel.
- Two delegation strategies are exclusive: `--status todo` child issue (parallel via different assignees) OR `@mention` on same issue (serial). Never both for the same work — that double-fires the worker.

**Scope discipline:** v0.2.0 is **purely additive**. v0.1.0 paths (single-agent discover→plan→execute→verify→report, HITL blocked protocol, persistence loop, verification Iron Law) remain unchanged. New behavior triggers only when the leader marker is present in the on-disk CLAUDE.md the daemon just wrote.

## 3. Guardrails

### Must Have
- v0.1.0 single-agent path **unchanged** in behavior; squad logic gates on marker presence in CLAUDE.md and falls through otherwise.
- Leader workflow includes the **mandatory** `multica squad activity` call at every turn end; the leader skill writes a marker file before exit; `hooks/stop.sh` detects skipped calls.
- `no_action` outcome ⇒ exit silently. **No comment is permitted whose only purpose is to announce no_action / acknowledgement.**
- Mention markdown rendered exactly `[@Name](mention://agent/<uuid>)`; roster format documented as the only valid syntax.
- Delegation strategy choice is explicit (one of: child-issue parallel | @mention serial), never both.
- Member HITL must attempt leader-tier first when the squad context exists; only fall through to human-tier when leader cannot resolve OR after 3 leader bounces with the same `question_id`.
- All new skill files ≤250 lines; mention all CLI commands via `<<cli:*>>` anchors (v0.1.0 rule).
- All new shell logic must be `shellcheck`-clean and pass `bash -n`.
- The string `## Squad Operating Protocol` is the leader contract marker; it must be declared as a shell constant `SQUAD_PROTOCOL_MARKER` in **both** `hooks/session-start.sh` (Step 5) and `hooks/stop.sh` (Step 6). Drift guard checks exactly these two constant declaration sites and rejects any additional occurrence elsewhere.

### Must NOT Have
- No environment-variable role detection (`$MULTICA_IS_LEADER` / `$MULTICA_SQUAD_ID`). Sole signal is the marker text in the on-disk CLAUDE.md. (Source: daemon.go:2260, prompt.go:145.)
- No fan-out heuristic that picks "both child issue AND @mention" for the same work.
- No leader attempting actual implementation in the squad path (only when roster is empty — documented exception).
- No member writing `[HITL:human]` directly when a leader is present in the roster; must try `[HITL:leader]` first.
- No leader HITL **reply** comment that contains an `@mention` to the same member who just escalated. (Reply is a directive; daemon already re-triggers the member via `on_comment`. Re-mentioning double-fires.)
- No new env vars from the plugin side that multica daemon hasn't promised. `MULTICA_WORKDIR` is the only env var read; it is already standard.
- No edits to v0.1.0 verification.md / systematic-debug.md / hitl-protocol.md (squad-tier is a layered protocol on top, not a rewrite).

### Step 3 — Create skills/core/squad-leader-workflow.md (M, 1d)

**File:** `/home/peter/project/research/multica-agent-plugin/skills/core/squad-leader-workflow.md`

**Required sections (in order):**

1. **Header + Iron Law**
   ```
   # squad-leader-workflow — Coordinate, Don't Execute
   ## Iron Law
   You are a leader. Your job is to delegate and record. You do not implement.
   ```

2. **Role detection**: one-paragraph confirmation rule — agent sees the protocol marker in its runtime config file (prose reference, no literal string here either — the literal is owned by session-start.sh). The hook will have already added a `Squad Role: LEADER` line to additionalContext when this skill is loaded; trust that signal.

3. **Roster reading**: how to parse the `## Squad Roster` block injected after the protocol. Roster rows are literal `[@Name](mention://agent/<uuid>)` markdown — copy them verbatim into delegation comments. **Never hand-author a mention link**; only paste from the roster.

4. **Per-turn workflow (mandatory)**
   - Phase 1 (discover): `<<cli:issue.get>>`, `<<cli:issue.comment.list.thread>>` (use the threaded form to scope context to the trigger thread). Read the latest trigger.
   - Phase 2 (decide): one of:
     - delegate → pick a member, draft a terse comment
     - escalate → @mention the human reporter
     - no_action → e.g. trigger was a member ack
   - Phase 3 (act):
     - delegate via Strategy A or B (see below) — never both for the same work
     - or set `<<cli:issue.status>>` to `blocked` and write `[HITL:human]` (see Escalation)
   - Phase 4 (record): **MANDATORY** `<<cli:squad.activity>>` with outcome `action|no_action|failed` and a one-line `--reason`. Skipping this call leaves the timeline incomplete and is a contract violation. After the CLI returns 0, **write the marker file** `${MULTICA_WORKDIR}/.multica/state/<issue-id>/squad-activity.marker` (just `mkdir -p` + `touch`) so `hooks/stop.sh` can verify this turn audited correctly. Issue id is the same uuid the daemon set when claiming the task; if the env var is not present, skip the marker write (stop.sh treats absent env as "not a leader turn").
   - Phase 5 (exit): end turn immediately after activity is recorded. No further tool calls.

5. **Delegation Strategy A — Child issue (parallel)**
   - When: tasks are independent and can run concurrently.
   - How: `<<cli:issue.create.child>>` (substitutes to `multica issue create --title "..." --status todo --assignee-id <uuid> --parent <parent-issue-id>`). The assignment IS the trigger; do not also @mention on the parent.
   - Concurrency reality: different assignee agents run in parallel. Same agent across N child issues is capped by that agent's `max_concurrent_tasks` (default 6). For capacity-aware delegation see the Concurrency Edge Cases section below.

6. **Delegation Strategy B — @mention on parent issue (serial)**
   - When: tasks have ordering dependencies, or the work is too small to justify a child issue.
   - How: post a comment on the parent issue via `<<cli:issue.comment.add>>` containing the roster's literal mention markdown for the chosen member. Wait for their reply (the daemon will re-trigger you).
   - Do not @mention the same member you already child-assigned — that double-fires them.

7. **Strategy decision matrix**
   ```
   Tasks independent + capacity available  → Strategy A (parallel child issues)
   Tasks must be sequenced                  → Strategy B (serial @mentions)
   Task is too small for a child issue      → Strategy B
   Same work assigned via both              → CONTRACT VIOLATION — pick one
   ```

8. **Concurrency edge cases**
   - **Capacity probe**: before fanning N child issues out to the same agent, expect at most 6 to be picked up immediately (default `max_concurrent_tasks`); the rest queue invisibly with no fairness guarantee. If parallelism matters, prefer fan-out across different agents.
   - **Mixed-batch (A + B)**: it is acceptable to use Strategy A for some sub-tasks and Strategy B for others in the same turn IF and ONLY IF every member is touched by at most one strategy. Same-member-via-both is the violation, not "both strategies present."
   - **Self-assignment**: do not create a child issue assigned to yourself in the squad path; that defeats the leader role. The empty-roster exception (section 11) covers the legitimate self-execution case.

9. **HITL reply discipline** (ping-pong guard — Critic M3)
   - When you read a `[HITL:leader] question_id=<uuid>` comment from a member, your reply is a **directive comment**, NOT a fresh delegation. **Do not include any `@mention` to the asking member in the reply body.** The daemon's `on_comment` trigger already re-fires the member when you post; an explicit mention is a second trigger that burns a turn and risks an infinite loop.
   - Carry the same `question_id` through your reply (e.g. `Re: question_id=<uuid> — <directive>`). If the same `question_id` arrives back from the same member more than once (i.e. the member's retry hit the same ambiguity), it counts toward the 3-strike rule below.
   - **3-strike escalation:** if a single `question_id` has been bounced back to you three times without resolution, your turn-3 action is NOT another directive — it is `<<cli:issue.status>>` to `blocked` plus a `[HITL:human]` comment (per v0.1.0 hitl-protocol.md) escalating to the workspace owner. Then record `<<cli:squad.activity>> action --reason "escalated to human after 3 leader bounces on question_id=<uuid>"`.

10. **Squad activity contract (verbatim)**
    - Every turn ends with exactly one `<<cli:squad.activity>>` call.
    - `outcome=action`: you wrote a delegation comment, created a child issue, or escalated to a human.
    - `outcome=no_action`: trigger required no response. Exit silently. **No comment.**
    - `outcome=failed`: you hit an error you cannot resolve (CLI failure, missing capability). Write a `[HITL:human]` comment then record `failed`.
    - `--reason` is a single short clause for the timeline audit trail.
    - On success the CLI prints `Squad evaluation recorded: <outcome> (issue <display>)` to stderr — this is the success signal; absence of that line means the call failed and you must retry or record `failed` and escalate.

11. **Tracking & rollup**
    - To find delegated child issues: `<<cli:issue.list>>` filtered by parent or by assignee.
    - When a member posts a result on a child issue, daemon re-triggers the leader (comment-triggered path).
    - On rollup-ready trigger: optionally summarize child outcomes into a parent comment, then `<<cli:squad.activity>> action`.

12. **Empty-roster exception**
    - If the roster summary injected by the session-start hook says "(none — you are the only member of this squad)", the leader may execute the work itself (single-agent fallback). Still record `<<cli:squad.activity>> action --reason "no other members; executing directly"`.

13. **Daemon-Safe Notes**
    - No sleep, no polling. Each turn is a single short process invocation.
    - Re-trigger is owned by the daemon (assignment, child completion, or new comment).
    - HITL escalation to human uses the v0.1.0 `hitl-protocol.md` template with the additional `[HITL:human]` tag prefix.

**Content rules:**
- ≤ 250 lines (bumped from 230 to fit new sections 8 + 9).
- No literal CLI commands outside `<<cli:*>>` anchors.
- Mention syntax appears exactly once as a copyable example.
- No env-var references EXCEPT `${MULTICA_WORKDIR}` (already standard; documented in CLAUDE.md).
- No literal `## Squad Operating Protocol` substring.

**Acceptance:**
- File exists at the specified path.
- `wc -l skills/core/squad-leader-workflow.md` reports ≤ 250.
- `grep -c "Iron Law" skills/core/squad-leader-workflow.md` returns ≥ 1.
- `grep -E "squad activity|<<cli:squad.activity>>" skills/core/squad-leader-workflow.md` returns ≥ 5 occurrences.
- `grep -c "Strategy A\|Strategy B" skills/core/squad-leader-workflow.md` returns ≥ 4.

### Step 4 — Create skills/core/squad-member-workflow.md (M, 0.75d)

**File:** `/home/peter/project/research/multica-agent-plugin/skills/core/squad-member-workflow.md`

**Required sections:**

1. **Header + role detection**
   - You are a member if: (a) your trigger comment includes a `[@YourName](mention://agent/<your-uuid>)` from a leader on a squad-tagged issue, OR (b) you are the assignee of a child issue created by a leader.
   - Your runtime config will NOT contain the protocol marker. Your role flag is purely contextual (the trigger).
   - The standard 5-Phase Workflow applies. This skill only modifies the HITL escalation path.

2. **Reading the task context**
   - When triggered by comment: `<<cli:issue.comment.list.thread>>` (substitutes to `multica issue comment list <issue-id> --thread <comment-id> --tail 30 --output json`) to read just the delegation thread.

3. **HITL two-tier protocol** (core addition over v0.1.0)
   ```
   When HITL trigger conditions fire (v0.1.0 hitl-protocol.md applies):

   Tier 1 — escalate to leader (try first):
     Write `[HITL:leader] question_id=<uuid>` comment, mention the leader using
     the `[@LeaderName](mention://agent/<leader-uuid>)` link captured from the
     delegation. Set status `blocked`. Exit.

   Tier 2 — fall through to human (only if leader cannot resolve):
     After leader replies "escalate to human" OR if the leader role is absent
     on the issue, write `[HITL:human]` per v0.1.0 hitl-protocol.md. Set status
     `blocked`. Exit.
   ```

4. **Triage decision (when to use which tier)**
   - Tier 1 (leader) covers: routing decisions ("should I do step 2 first?"), member-swap requests, scope clarifications.
   - Tier 2 (human) covers: missing credentials, destructive operations beyond the leader's authority, architectural decisions the leader explicitly punts on.
   - **3-strike failure** (mirrors the leader skill — Critic M3): the FIRST attempt is `[HITL:leader]`. If the leader replies with a directive ("try again with X") and that retry fails, **on the 3rd attempt with the same `question_id`** the member writes `[HITL:human]` directly. The leader-side 3-strike rule (leader skill section 9) is the redundant safety net; both sides counting independently means a stuck leader cannot trap a member in an infinite bounce.

5. **Reply path on completion**
   - Post final result via `<<cli:issue.comment.add>>` on the same issue (parent or child) where you were triggered.
   - Set status: child issue → `done` (this auto-notifies the leader); parent comment-triggered → no status change (leader will close).
   - **Do not @mention the leader on completion** unless they explicitly asked you to. Auto-notification via status / comment is enough; an unnecessary mention re-triggers the leader and burns a turn.

6. **Daemon-Safe Notes**
   - No sleep / polling. Member exit pattern identical to v0.1.0 (set status, exit).
   - On HITL bounce-back from leader, daemon re-triggers via `on_comment`; resume Phase 1 with the new comment as trigger.

### Step 6 — Extend hooks/stop.sh with leader audit guard (S, 0.5d)

**File:** `/home/peter/project/research/multica-agent-plugin/hooks/stop.sh`

**Background (Critic M2):** the contract says every leader turn ends with `multica squad activity`. A leader skill that forgets the call leaves the timeline silently incomplete — the failure mode in Pre-mortem Scenario C. We need a passive detector.

**Approach (no daemon-side change required):**
1. The leader skill (Step 3, Phase 4) writes a marker file `${MULTICA_WORKDIR}/.multica/state/<issue-id>/squad-activity.marker` immediately after a successful `squad activity` call.
2. `hooks/stop.sh` checks: if CLAUDE.md contains the leader marker AND no `squad-activity.marker` exists for any issue id in the current turn's state directory, write a warning file `${MULTICA_WORKDIR}/.multica/state/squad-audit-warning` containing a one-line message. Session-start picks it up on the next session and surfaces it as a high-priority hint.
3. We do NOT block exit (Stop hook return non-zero would loop Claude in older versions). We only record evidence.

**Pseudo-code for the new block in `stop.sh`** (append; do not refactor):
```bash
SQUAD_PROTOCOL_MARKER="## Squad Operating Protocol"
claude_md="${MULTICA_WORKDIR:-.}/CLAUDE.md"
state_root="${MULTICA_WORKDIR:-.}/.multica/state"

if [[ -f "$claude_md" ]] && grep -qF "$SQUAD_PROTOCOL_MARKER" "$claude_md"; then
  # Leader turn. Did any squad-activity.marker land this session?
  # Heuristic: at least one marker file under state_root whose mtime is within
  # the current session window. Session start time is exported by session-start.sh
  # as ${MULTICA_SESSION_STARTED_AT_EPOCH}; fall back to "any marker exists" if
  # that env var is missing (older session-start build).
  found=0
  while IFS= read -r -d '' f; do
    if [[ -n "${MULTICA_SESSION_STARTED_AT_EPOCH:-}" ]]; then
      mtime=$(stat -c %Y "$f" 2>/dev/null || stat -f %m "$f" 2>/dev/null || echo 0)
      if (( mtime >= MULTICA_SESSION_STARTED_AT_EPOCH )); then
        found=1; break
      fi
    else
      found=1; break
    fi
  done < <(find "$state_root" -name 'squad-activity.marker' -print0 2>/dev/null || true)

  if (( found == 0 )); then
    mkdir -p "$state_root"
    printf 'Leader turn ended without recording squad activity. Add the squad activity call before exit next turn.\n' \
      > "$state_root/squad-audit-warning"
  else
    # Clear any stale warning from a prior gap.
    rm -f "$state_root/squad-audit-warning"
  fi
fi
```

**`session-start.sh` companion change** (lightweight, same step):
- Export `MULTICA_SESSION_STARTED_AT_EPOCH=$(date +%s)` early in the hook so stop.sh can scope its marker search.
- If `${MULTICA_WORKDIR:-.}/.multica/state/squad-audit-warning` exists, prepend the warning to `context_parts` (highest priority) and delete the warning file so it only surfaces once.

**Content rules:**
- Both `session-start.sh` and `stop.sh` declare `SQUAD_PROTOCOL_MARKER` locally (two declarations, but both files are the marker's authoritative location — the drift guard allows the marker to appear in `hooks/session-start.sh` and `hooks/stop.sh`). Update the drift-guard whitelist to permit exactly those two declarations and no others.
- No external network calls. No daemon-side modifications.
- `stop.sh` MUST NOT exit non-zero on a missing marker — Stop hooks with non-zero exit re-loop Claude in some Claude Code versions; we want passive detection, not enforcement.

**Acceptance:**
- `bash -n hooks/stop.sh` passes.
- `shellcheck hooks/stop.sh` zero error-level findings.
- Drift guard update: `grep -rF "## Squad Operating Protocol" .` returns exactly 2 lines (one in each hook). Update the test in Step 7 expected-count from 1 to 2.
- Positive test: simulate a leader turn that touches the marker file → no warning written.
- Negative test: simulate a leader turn without touching the marker → warning file written; subsequent session-start picks it up and consumes (deletes) it.
- Non-leader regression: simulate a single-agent turn (no marker in CLAUDE.md) → stop.sh does nothing.

## 8. Pre-mortem (DELIBERATE)

**Scenario A — Leader silent failure 3 weeks in**
multica daemon ships a minor version bump that renames `## Squad Operating Protocol` to `## Squad Protocol`. Leader detection silently falls through to single-agent path. Leaders begin executing work themselves, defeating the squad.
**Mitigation now:** (1) marker is pinned as `SQUAD_PROTOCOL_MARKER` constant in two files only (session-start.sh + stop.sh); (2) `grep -rF` drift guard CI step asserts exactly 2 lines repo-wide; (3) smoke scenario 5 asserts the leader hint appears in additionalContext when the fixture marker is present; if the daemon changes the heading, the smoke fixture (which we'd update to match) breaks against the constant — instant detection.

**Scenario B — HITL ping-pong loop**
Member writes `[HITL:leader]`, leader misreads it as a delegation prompt and re-mentions the same member back with the same question. Member re-fires, posts the same HITL again. Infinite loop, ~$N tokens per cycle.
**Mitigation now:** (1) leader skill section 9 explicitly bans `@mention` in HITL replies; the reply is a directive, not a re-delegation — daemon's `on_comment` already re-triggers the member; (2) `question_id` discipline carries from v0.1.0 hitl-protocol.md — same `question_id` ⇒ leader recognizes the bounce; (3) leader 3-strike + member 3-strike: same `question_id` arrives 3 times ⇒ leader escalates to `[HITL:human]` AND member escalates to `[HITL:human]` directly — both sides have an independent cap so a stuck leader cannot trap a member.

**Scenario C — squad activity skipped under stress**
Leader hits a CLI failure mid-turn (e.g. `multica issue comment add` returns non-zero). Leader process crashes or returns early without calling `multica squad activity`. The timeline shows the delegation comment but no audit entry; multiple leaders hitting this same path silently break the audit guarantee.
**Mitigation now:** (1) leader skill explicit Iron Law: `squad activity` is the last call of every turn, with the marker-file write immediately after; (2) `outcome=failed --reason "<cli-error>"` documented as the catch-all for partial failures; (3) `hooks/stop.sh` (Step 6) writes a `squad-audit-warning` file when the leader marker is in CLAUDE.md but no `squad-activity.marker` was written this session; session-start consumes the warning on next turn and surfaces it as high-priority context — the next leader run sees "you skipped squad activity last turn, do not skip it again."

## 10. ADR (Embedded Summary)

- **Decision:** Add Squad-aware behavior as a layered overlay on top of v0.1.0 paths, with role detection driven by **on-disk marker text** in `${MULTICA_WORKDIR}/CLAUDE.md` (the file daemon writes via `runtime_config.go:101-106`) and two new skill files (`squad-leader-workflow.md`, `squad-member-workflow.md`) that the existing AGENTS.md indexes. Extend the session-start hook to detect the marker and inject a roster summary; extend the stop hook to passively audit leader turns for missing `squad activity` calls.
- **Drivers:**
  1. Multica daemon already commits to text-marker detection in two places (`daemon.go:2260`, `prompt.go:145`); building the plugin around env vars would diverge from a contract the daemon already holds.
  2. The mandatory `multica squad activity` call defines the squad audit trail; any leader path that omits it breaks the platform invariant. Both an in-skill marker-file write and a stop-hook audit are needed to make skip behavior observable.
  3. v0.1.0 paths are already proven; the lowest-risk integration is purely additive, gated on marker presence in a file the daemon already writes.
- **Alternatives considered:**
  - **A — Env-var detection (`MULTICA_IS_LEADER=true`).** Pros: simple, no file parsing. Cons: multica daemon doesn't set it (verified `daemon.go:2260` + `prompt.go:145`); requires upstream change; couples plugin to a contract daemon hasn't committed to. **Rejected.**
  - **B — Rewrite multica-workflow.md to be squad-aware.** Pros: single skill, no role detection. Cons: bloats the v0.1.0 file past the 150-line context-rot guard; conflates two roles (leader coordinates / member executes) into a "god skill" with internal `if leader / if member` branches the agent must parse every turn; selective disabling harder. **Rejected.**
  - **C — Detect via a virtual `.multica/briefing.md` file the plugin would have asked daemon to write.** Pros: a dedicated file is cleaner than parsing CLAUDE.md. Cons: requires a daemon-side change (and a release coordination dance); daemon already writes CLAUDE.md with the embedded marker, so adding a second file is redundant work; risk of the second file falling out of sync with CLAUDE.md. **Rejected** in favor of using the file daemon already writes.
  - **D — Layered overlay with on-disk CLAUDE.md detection (selected).**
- **Why chosen:** Option D matches the daemon's actual role-detection mechanism, requires zero daemon-side changes, preserves v0.1.0 regression guarantees, and degrades to single-agent behavior when the file or marker is absent. It is the only option that respects the "AGENTS.md ≤ 150 lines + skills small and on-demand" context-rot principle inherited from v0.1.0 while also being deployable today.
- **Consequences:**
  - Plugin's leader detection is coupled to (a) the daemon writing CLAUDE.md and (b) the marker heading text. R1 + R2 manage these.
  - Smoke tests gain three new scenarios with a non-trivial mock requirement.
  - Leader path forbids self-execution except for empty rosters; this constrains future "self-executing leader" features (acceptable for v0.2.0).
  - The `SQUAD_PROTOCOL_MARKER` constant must live in exactly two files (session-start.sh + stop.sh). Future additions of the literal string in any other file must update the drift-guard test and pass code review.
  - Future v0.3.0 can layer cross-member coordination (broadcast comments, leader-of-leaders, etc.) without touching v0.2.0 paths.
- **Follow-ups:**
  - Consider asking multica to expose a `multica daemon caps` probe that emits the protocol-text version once daemon supports it.
  - Member-side handling for orphaned squads (leader removed mid-flow) — currently routes through `[HITL:human]`.
  - `multica squad activity log` summarization in the leader rollup phase (if/when CLI exposes a `log` subcommand).
  - Replace the heuristic 3-strike count with a structured `question_id` history once daemon adds such a field.

## RALPLAN-DR Summary

### Principles (5)
1. **Daemon contract is the source of truth.** Role detection mirrors `daemon.go:2260` + `prompt.go:145` (text presence in CLAUDE.md the daemon writes), not invented env vars.
2. **v0.1.0 is sacred.** Squad logic is additive overlay; single-agent path is unmodified and must regression-test green.
3. **Coordinate, don't execute.** A leader without an empty roster never implements; this is the squad's reason to exist.
4. **Audit by default.** `multica squad activity` is mandatory every turn — including `no_action` — because the timeline is the squad's primary observability surface. Skipped calls are detected by a passive stop-hook guard.
5. **Silence is a feature.** `no_action` exits silently; HITL escalates leader-first; HITL replies omit `@mentions`; each unnecessary comment costs a re-trigger.

### Decision Drivers (top 3)
1. **Daemon already committed to text-marker role detection AND writes CLAUDE.md every claim** — plugin must align to avoid divergence (R1 + R2 + Scenario A in pre-mortem).
2. **The `squad activity` mandatory contract is the most fragile invariant** — any path that skips it silently breaks audit; both an in-skill marker + a stop-hook passive audit are needed (Scenario C in pre-mortem).
3. **Additive overlay vs. rewrite** — v0.1.0 has 4 passing smoke scenarios; preserving them at zero diff is cheaper than re-validating a unified workflow.

### Viable Options

#### Option D — Layered overlay with on-disk CLAUDE.md marker detection (**recommended**)
- Pros: matches daemon contract exactly; uses the file daemon already writes (no daemon-side change); additive, no v0.1.0 regression risk; smoke-testable; degrades cleanly when CLAUDE.md is absent
- Cons: requires drift-guard discipline for the marker constant; the leader-audit guard is passive (warning, not enforcement)

#### Option B — Unified squad-aware multica-workflow.md
- Pros: single skill, no role detection branching at the AGENTS.md level
- Cons: violates the 150-line context-rot guard; conflates 3 paths (leader / member / solo); leader and member workflows are genuinely different (coordinate vs. execute) so unifying them produces a "god skill" with internal `if leader / if member` branches the agent must parse on every turn

#### Option C — Detect via a dedicated `.multica/briefing.md` file (rejected for redundancy)
- Pros: a separate file dedicated to briefing is conceptually cleaner
- Cons: daemon already writes the marker into CLAUDE.md as part of `## Agent Identity`; adding a second file is redundant, creates synchronization risk, and requires a daemon-side change; option D needs no daemon change

### Recommendation

**Option D.** It aligns with the daemon's actual mechanism, requires zero daemon-side changes, preserves v0.1.0 regression guarantees, and degrades observably when the marker is missing. Options B and C are explicitly invalidated:
- **B** is invalidated by the context-rot rule and by the genuine semantic difference between leader and member workflows.
- **C** is invalidated by redundancy: daemon already writes the marker into CLAUDE.md (verified at `runtime_config.go:101-106` + `137`); inventing a parallel file creates synchronization risk and a needless daemon coupling.
