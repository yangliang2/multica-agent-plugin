## 执行摘要

本次审视从用户研究、产品设计、易用性三个维度对 multica-agent-plugin v0.4.0 进行全面专业审视。审视结果发现：

- **🔴 致命问题 8 条**：涉及核心功能可靠性（完成信号机制、CLI ABI 断裂、并发模型未落地）及用户安装的根本性摩擦
- **🟠 严重问题 18 条**：涉及产品定位混乱、错误处理缺失、认知负荷过重等
- **🟡 中等问题 15 条**：涉及文档一致性、跨平台兼容性、配置复杂度等
- **🟢 轻微问题 若干**：见各章节

**最紧迫结论**：当前 v0.4.0 不具备声称的"production-ready"特征。完成信号 cross-check 未实现、CLI ABI 双轨断裂、并发 flock 仅是 spec 文档、hooks 全静默失败——这四条致命问题使核心工作流可靠性存疑。

## 一、用户研究视角 Review

> **审视人 A** | **日期**：2026-05-24

### 核心发现摘要

1. **🔴 用户身份根本性混淆**：产品宣称同时服务 AI agent 与人类开发者/团队，但二者认知模式、错误恢复能力、阅读路径几乎完全不同。双受众文档（USAGE.md §1-2 vs §3-4）只解决了写作组织问题，没有解决产品定位问题。整个项目找不到"我们优先为 X 用户优化，因此牺牲了 Y"的清晰取舍声明。
2. **🔴 价值主张完全缺失**：README 全篇列功能，不列收益。读完无法回答"我为什么需要这个插件，而不是直接用 Claude Code + Multica CLI"。
3. **🔴 首次用户旅程有致命断点**：install.sh 不能完成可用安装，必须手动 export `MULTICA_PLUGIN_ROOT` 并重启 Claude Code，hooks 失败默认静默（exit 0 by design）。"安装成功"与"工作正常"之间没有任何反馈信号。
4. **🟠 心智模型与目标用户工作方式严重错位**：5-phase 状态机、HITL 三段式 comment 模板、Squad 双轨委派、3-strike 计数、context budget 35%/25% 阈值——这是给纪律严明的 agent 设计的，不是给人类用户设计的。但人类用户必须理解这一切才能诊断问题、写好 issue、判断 agent 卡在哪。对"使用 Multica 的开发者"完全没有铺垫和缓冲。
5. **🟠 核心假设未经用户验证**：5 phase 而非 4 或 7、HITL 阈值 3 次、squad leader/member 模式——没有任何用户访谈、问卷、A/B 数据佐证。所有设计决定看起来都是"内观+架构推导"。

### 详细审视意见

#### 1.1 🔴 "AI agent 是用户"是模糊话术，掩盖设计冲突

**表现**：README 说"AI agents as first-class citizens"；USAGE §3 把 agent 称为 audience；但 agent 不会购买、不会流失、不会反馈。AGENTS.md 第 1 行写"No human is at the keyboard"，但 USAGE §1 又说"for humans setup"——同一个插件，到底有没有人在场？

**影响**：当人类体验和 agent 操作正确性冲突时（如 HITL 模板对 agent 是约束，对 reviewer 是阅读负担），项目无法判断该优先谁。

**建议**：明确区分三类角色——plugin operator（运维/安装人）、issue reviewer（看 comment 做 HITL 回复的人）、agent（被插件约束的执行体）。每个 feature 标注主要受益者。

#### 1.2 🟠 "需要协调多 agent 的技术负责人/工程团队"在文档中几乎不存在

**表现**：宣称的第三类用户在 README、USAGE 中几乎没有出现。Squad coordination 只描述 agent 间的协议，没有描述"技术负责人如何观察 squad 健康度、何时介入、如何 debug 卡住的 squad"。

**建议**：要么删除这个用户群体，要么补一节"For Tech Leads: Observability & Intervention"。

#### 1.3 🟡 Troubleshooting 每条修复步骤缺少角色标注

**表现**：USAGE §5 troubleshooting 里"Fix: read loop.json to find current phase"——这个修复者是谁？运维？产品经理？难度与目标用户能否执行不匹配。

**建议**：每条 fix 上标注"who can do this: ops / reviewer / dev"。

#### 2.1 🔴 README 没有 elevator pitch

**表现**：README 前 50 行讲"What This Is"和"Features"，读完无法判断 ROI——用了之后 agent 失败率下降多少？issue 平均时长缩短吗？

**建议**：开头加 3 句话——"问题 / 解决方式 / 最小可观察改善"。例如："Without this plugin, agents drift into loops, falsely claim 'done', and break squad coordination. This plugin enforces verification, structured HITL, and squad protocols at the hook level, so daemon-spawned sessions terminate predictably."

#### 2.2 🟠 强约束缺少"为什么需要"的真实失败案例

**表现**：USAGE 列出 Iron Law 1/2/3，每条只有一句"Why: ..."，没有"没有 Iron Law 时的失败案例"。

**建议**：每条 Law 配一个脱敏真实失败案例。如"Last quarter, 23% of agent sessions claimed done without running tests; 8% of those broke prod."

#### 2.3 🟠 v0.4.0 知识管理收益完全未量化

**表现**：CHANGELOG v0.4.0 没说"用了 learnings.jsonl 之后重复犯同类错误的次数下降了 X%"，用户没法判断该不该启用 git commit 知识同步。

**建议**：附"the learning loop expectation"并提供观察收益的指标。

#### 3.1 🔴 安装到首次工作之间存在"沉默失败"鸿沟

**表现**：install.sh 只做 3 件事（检查 multica CLI、merge hooks.json、打印提示）；`MULTICA_PLUGIN_ROOT` 必须手动加入 shell profile；hooks 失败时 by design exit 0（KNOWN-LIMITATIONS.md:21-23）；首次启动找不到 MULTICA_PLUGIN_ROOT 会怎样文档没说。

**建议**：(a) 加 `bash install.sh --verify` 子命令确认 hooks 生效；(b) install.sh 提供选项自动写入 shell profile；(c) hooks 失败写结构化错误到 `.multica/install-health.log`。

#### 3.2 🟠 "Restart Claude Code" 指令对 daemon 场景没意义

**表现**：install.sh:93 说 "Restart Claude Code for hooks to take effect"，但项目背景说 agent 是被 daemon spawn 的——daemon spawn 新 session 时是新进程，不需要"重启 Claude Code"。

**建议**：区分两种安装目标——本地开发者验证 vs daemon worker 部署。

#### 3.3 🟠 没有"卸载后系统状态"承诺

**建议**：uninstall 提供 dry-run，明确列出会删除什么、保留什么。

#### 3.4 🟠 没有"hello world" demo

**建议**：加 "Quickstart" 章节——一个最简 issue 跑通 5 phase 并展示 comment trail。

#### 4.1 🔴 5-phase 状态机对人类 issue reviewer 是认知负担

**表现**：reviewer 看 comment 时序时，需要 reconstruct 5 个 phase，但 comment 本身没有 phase 标记（只有 [verification]/[HITL]/[checkpoint]），不知道 agent 现在在哪个 phase。

**建议**：每条 phase-transition comment 加 `[phase] discover→plan` 结构化前缀，便于 reviewer 扫读。

#### 4.2 🟠 HITL 模板对 reviewer 的认知模型假设过强

**表现**：模板要求 A/B 选项回复，但真实人类常发起反问、给第三选项。当前协议没有定义 agent 收到反问的处理路径。

**建议**：明确"如果 reviewer 回复不是 A/B 之一，agent 必须再发 HITL 求确认"，在模板里增加合法的"ask back"路径。

#### 4.3 🟠 Squad comment 无角色标注，reviewer 难以理解多 agent 因果链

**建议**：每条 squad comment 加 `[squad-role:leader]` 或 `[squad-role:member name=X]` 前缀。

#### 4.4 🟠 Context budget checkpoint 和 actionable HITL 混在一起污染 reviewer 视野

**表现**：context budget critical 时 agent 会发 `[checkpoint]` comment + 设 blocked。人类 reviewer 收到这条消息后能做什么？无法"给 agent 更多 context"。

**建议**：区分 internal-state checkpoint（写 metadata，不发 comment）和 actionable HITL（发 comment 给 reviewer）。

#### 4.5 🟡 "3-strike" 阈值的认知合理性未说明

**建议**：给出数据支撑（"3 次失败后继续尝试成功率<5%"），或提供配置项。

#### 5.1 🔴 假设"multica 用户接受 agent 是异步 issue worker 心智模型"未验证

**建议**：访谈 3-5 个真实 multica 用户，问"你最近一次 agent 卡住时，你期望它做什么？"

#### 5.2 🟠 假设"人类 reviewer 会及时回复 HITL"无 SLA 保证

**表现**：HITL 协议禁止 agent 自实现 timeout，把责任完全推给 daemon reaper，但 daemon reaper SLA 文档没说。

**建议**：明确"daemon reaper 在 X 小时后会做什么"，或由 plugin 端记录 HITL 的 wait age。

#### 5.3 🟠 假设"squad leader 总能不写代码完成协调"未验证

**建议**：squad 模式应有"任务规模阈值"——小于 N 行的任务允许 leader 直接做。

#### 5.4 🟠 假设"JSONL + git commit 是合理知识同步通道"有并发冲突风险

**表现**：多机并发同 issue 会产生 merge conflict。

**建议**：用独立 branch / orphan ref（`refs/multica/learnings`），或中心化 store（Multica 服务端 API）。

#### 6.1 🔴 失败模式 A：用户装好但 hooks 不生效，不知情

**触发**：MULTICA_PLUGIN_ROOT 没 export、daemon 启动环境没继承用户 shell。**当前无任何失败信号。**

**建议**：提供 `multica-plugin doctor` 子命令，检查 env、hooks 注册、cli 版本、git 状态、python3。

#### 6.2 🔴 失败模式 B：agent 反复 HITL 同一个问题

**触发**：reviewer 回复格式不被 agent 识别，或 agent 重启后没找到 question_id。**"已回答"的判定算法文档未说明。**

**建议**：每个 HITL question_id 的状态机要可见——pending / answered / acknowledged。

#### 6.3 🟠 失败模式 C：context budget blocked，reviewer 不知道怎么解

**建议**：blocked 类型应有处置 playbook，每种 reason 配"recommended human action"。

#### 7.1 🟡 双受众文档结构是反模式

**建议**：拆成 `docs/HUMAN-GUIDE.md` 和 `docs/AGENT-CONTRACT.md`。

#### 7.2 🟡 README "your-org" 占位符未替换

**建议**：发布前换成真实 org，否则产品可信度受损。

#### 7.3 🟡 README 安装段与 USAGE.md 安装段矛盾

**建议**：README 只指向 USAGE.md 单一来源。

## 二、产品设计视角 Review

> **审视人 B** | **日期**：2026-05-24

### 核心发现摘要

1. **🔴 致命：hooks 实际调用的 CLI 参数与文档 ABI 不一致**——`hooks/stop.sh` 和 `hooks/pre-tool.sh` 用的是 `multica issue comment add --issue X --body Y` 和 `multica issue status set blocked`，但 `docs/abi/cli-outward.md` + `docs/cli-reference.md` + `tools/render-anchors.sh` 规定的是 `multica issue comment add <issue-id> --content "..."` 和 `multica issue status <id> <status>`。ABI 锁文件、anchor 渲染器、smoke 测试全部围绕文档化形态建立，但 hooks 实际运行的是另一种形态。**核心通信链路在自欺欺人。**
2. **🔴 致命：`<promise>DONE</promise>` 的完成信号机制 cross-check 未实现**——`hooks/stop.sh` 只检 `CLAUDE_TOOL_OUTPUT` 和 `MULTICA_OUTPUT_FILE` 是否含 DONE 字符串，完全没有交叉验证 `loop.json` 中所有 stories 的 `passes: true`。任何包含该字面量的输出（包括 agent 抄写文档）都会立即结束 loop。skill 文档里写的安全承诺与 hook 代码不匹配。
3. **🔴 致命：并发模型纸上谈兵**——`docs/concurrency-model.md` 用一整页规定 advisory lock，但 `grep -rn flock` 在整个仓库脚本中**零命中**。`stop.sh` 直接对 `loop.json` 做 awk 解析 + atomic_write，没有任何 lock 保护。这是已写但未实施的设计文档。
4. **🟠 严重：plugin/Multica/Claude Code 三方职责边界混乱**——plugin 自己做 git commit 知识同步、直接调 `multica squad activity failed`，同时强制 agent 不做 daemon 才有的事（timeout）。这个 plugin 实质是一个"伪 daemon"，但文档没明确这一点。
5. **🟠 严重：v0.4.0 知识管理三个新管道入口没有一个有验证的出口**——learnings 写入靠 LLM 自觉无 schema 校验；consolidation-prompt.txt 生成后无任何调度器执行；multi-machine git sync 假设多个未验证的前提条件。

### 详细审视意见

#### 1.1 🟠 Plugin 边界从未被显式定义

**表现**：`hooks/stop.sh` 在 DONE 路径上执行 `git commit`（daemon 该做的）；`stop.sh` squad 审计失败时直接调 `multica squad activity failed`（越权进 daemon 责任域）；AGENTS.md 禁止 agent 自实现 timeout，但 plugin 设计了 max_iterations=50 硬上限（等于实现了 sub-daemon watchdog）。

**建议**：README/AGENTS 第一段写清 plugin vs daemon vs Claude Code 的边界；git commit、squad activity failed 要么移除，要么由 daemon 显式授权后执行。

#### 1.2 🟡 Daemon inward ABI 缺失

**表现**：`MULTICA_ISSUE_ID`、`MULTICA_OUTPUT_FILE`、`MULTICA_WORKDIR`、`CLAUDE_TOOL_OUTPUT` 是 plugin 的输入契约，但 README 和 USAGE 只字未提 daemon 应如何注入。新 harness 接入必须看代码反推。

**建议**：在 `docs/abi/` 下新增 `daemon-inward.md`，列出"daemon 必须设置的 env var 列表 + 何时设置 + 默认值降级行为"。

#### 2.1 🔴 完成信号机制的设计漏洞（详细）

**表现**：`persistence-loop.md` 写道"The Stop hook cross-checks `loop.json` state"，但 `hooks/stop.sh:75-85` 实际代码只做字符串 grep，`L107` 直接进入 done 分支，**完全没有读 `loop.json`、验证所有 `stories[].passes == true`**。

**三个具体风险**：
1. agent 在注释/PRD 文档/commit message 里抄写 DONE 字面量 → 立即触发 done
2. `MULTICA_OUTPUT_FILE` 是累加写还是覆盖写——文档没说，前一次的 DONE 可能残留
3. `consolidation-prompt.txt` 模板本身含 `<promise>DONE</promise>` 字面量风险

**建议**：(a) stop.sh 必须读 loop.json 确认 `phase=="complete"` 且 `all(stories[].passes==true)`；(b) 用 nonce 化签名（`<<MULTICA_PROMISE_DONE:<issue_id>:<run_nonce>>>`），session-start 生成 nonce 并 export，stop hook 校验。

#### 2.2 🟠 HITL 状态机竞态——blocked 状态重启后的行为依赖 LLM 自觉

**表现**：`hitl-protocol.md` 说 "If restarted and finds status is already `blocked` with an open HITL, wait for human reply"——但 plugin 没有任何执行体强制这个，session-start 不读 issue status，agent 重启直接动手会双写 HITL。

**建议**：把"如果 status==blocked 且 last HITL 未被回复，立即 exit"作为 hook 级强制（PreToolUse gate 或 session-start exit）。

#### 2.3 🟡 CLI ABI 版本号三处互相矛盾

| 文件 | 声明版本 |
|------|----------|
| `KNOWN-LIMITATIONS.md:17` | multica >= 0.4.0 |
| `docs/abi/cli-outward.md:7` | multica >= 0.3.4 |
| `docs/abi/cli-outward.md:203-205` | multica >= 0.3.4 |

**建议**：单一来源——minimum version 放到 `capabilities/claude-code.json`，docs/code 都读这个字段。

#### 2.4 🟡 cli-reference.lock SHA-256 无防篡改作用

**表现**：`refresh-cli-reference.sh` 既生成 md 又生成 lock，任何人改完 md 重跑脚本就能"合法"绕过。

**建议**：改名为 `cli-reference.sha256`（去掉"lock"语义），或引入外部 attestation。

#### 3.1 🔴 anchor 体系与实际 CLI 调用断裂（双轨问题）

**表现**：
- **Anchor 轨**（skills 文档全用）：`<<cli:issue.comment.add>>` → `multica issue comment add <issue-id> --content "..."`
- **实代码轨**（hooks 实际跑）：`multica issue comment add --issue <id> --body <text>`

这意味着 render-anchors.sh 渲染出的命令和 hook 实际跑的命令**不同**；smoke test 通过 ≠ ABI 契约通过；新 harness 移植时看 skill 文档照抄会得到错误的 CLI 形态。

**建议**：hooks/*.sh 必须使用与 ABI 文档一致的参数形态；render-anchors.sh 应从 `docs/cli-reference.md` 自动构造 ANCHOR_MAP，而非硬编码。

#### 3.2 🟠 状态写入既有 atomic_write，又有非原子直写

**表现**：`hooks/stop.sh:113-116` 用 `sed | sed` pipe 后 atomic_write——sed 失败时变量为空串，atomic_write 会写空 loop.json（数据丢失）；`stop.sh:143` `awk ... > tmp && mv tmp notepad || rm tmp` 的短路写法，mv 失败时 rm 删 tmp 且不报错（静默丢数据）。

**建议**：所有 state 文件统一用 python3 解析+写回；shell 短路链改成显式 if/else。

#### 3.3 🟡 5 种 JSON 解析方式并存

`stop.sh` 用 `awk -F'"`，`session-start.sh` 用 awk + python3 混合，`curate-memory.sh` 用 python3，`install.sh` 用 python3，`session-start.sh` capability 字段用 jq。

**建议**：python3 已是 v0.4.0 软依赖，统一用 python3，下线其他路径。

#### 3.4 🟡 render-anchors.sh 的 squad anchor 缺失

**表现**：`squad-leader-workflow.md` 用 `<<cli:issue.create.child>>`；`squad-member-workflow.md` 用 `<<cli:issue.comment.list.thread>>`；但 render-anchors.sh 的 `ANCHOR_MAP` 里**没有这两个 anchor**——对 squad skill 运行 render-anchors 会直接 exit 2。CHANGELOG 写"三个新 anchor 增加"，render-anchors.sh 没更新。

#### 4.1 🟠 所有 multica CLI 调用都用 `|| true` 吞掉错误

## 三、易用性视角 Review

> **审视人 C** | **日期**：2026-05-24

### 核心发现摘要

1. **🔴 `MULTICA_PLUGIN_ROOT` 手动 export 是致命摩擦点**（`install.sh:84-86`）：只打印提示让用户自己改 shell profile，无自动化、无验证步骤。一旦遗忘，所有 hooks 变成路径解析失败的死命令（`${MULTICA_PLUGIN_ROOT}/hooks/stop.sh` → 字面字符串），且不会报错，留下损坏配置。
2. **🔴 hooks.json 占位符认知不一致**：`install.sh:52` 在 merge 前用 sed 替换为绝对路径，但 `CLAUDE.md:22-26` 给用户展示的是带占位符的 settings.json，造成认知错位。实际含义：移动 plugin 目录会让 hooks 失效，没有任何文档提示。
3. **🔴 22 处 `2>/dev/null` / `|| true` 全静默失败哲学使系统失去可调试性**：`stop.sh:4` 定义了 `die()` 但全文 0 处调用。multica CLI 鉴权过期、网络断、subcommand 不兼容时，agent 行为正常退出，checkpoint 评论永远不会出现在 issue 上，用户没有任何线索。
4. **🟠 概念负荷爆炸：成功使用需理解至少 12 个并行概念**：5-phase state machine + Iron Laws ×3 + HITL（两层）+ Squad（双角色）+ Persistence Loop + Subagent dispatch + Context budget + 三种 model routing + Notepad 三段 + Learnings 7 字段 schema + Squad activity audit + Working Memory 7 天 prune。USAGE.md 326 行无渐进路径。
5. **🟠 调试体验近乎为零**：无 log 文件、无 verbose 模式、无诊断命令、无 hook 启动成功反馈。Troubleshooting 5 个场景的修复方法基本是"读 loop.json、检查环境变量"——需要人工考古学。

### 详细审视意见

#### 1.1 🔴 MULTICA_PLUGIN_ROOT 手动 export 设计缺陷

**表现**：`install.sh:84-86` 仅 `echo` 提示用户编辑 shell profile；不检测 shell 类型；不提供 `--write-profile` 选项；不验证后续是否生效。安装脚本依然打印"installed successfully"（`install.sh:92`），用户以为装好了。

**建议**：(a) install.sh detect `$SHELL`，自动 append 到对应 profile；(b) hook 启动时若 env 缺失，写警告到 `.multica/state/install-broken.warn`，由 session-start 在下次启动时大字提示；(c) 提供 `bash install.sh --verify` 端到端探活。

#### 1.2 🔴 hooks.json 占位符的认知不一致

**表现**：`hooks/hooks.json` 内是字面 `${MULTICA_PLUGIN_ROOT}`；`install.sh:52` 用 sed 替换为绝对路径；但 `CLAUDE.md:22-26` 展示的是带占位符的片段。用户认为是"动态展开"，实际是"安装时固化"。`MULTICA_PLUGIN_ROOT` env var 在 hooks.json 触发链路上实际是 dead（`session-start.sh:197` 才真正读它）。

**建议**：(a) 文档明确说"settings.json 内是 install 时刻的绝对路径"；(b) 提供 `bash install.sh --relocate` 自动更新 settings.json；(c) 或让 hooks 真的使用占位符并由 thin shim 在 runtime 解析。

#### 1.3 🟠 install.sh 依赖检查不完整

**表现**：只检查 `multica`（line 23）和 `python3`（line 39）。`session-start.sh:202` 需要 `jq`（缺失时静默回退到默认 model routing），`stop.sh:120` 需要 `git`——这些缺失时静默跳过，用户不知道。

**建议**：install.sh 输出依赖矩阵：`[install] python3: OK | jq: MISSING (model routing will use defaults) | git: OK`。允许用户 informed-consent 后继续。

#### 1.4 🟡 install.sh 没有 dry-run / 没有备份 settings.json

**表现**：`install.sh:73-75` 直接覆盖 settings.json，没有备份。`set -e` + `printf '{}'` 组合可能在错误时覆盖已有配置。

**建议**：merge 前 `cp settings.json settings.json.bak-$(date +%s)`。

#### 1.5 🟡 README.md 与 USAGE.md 安装步骤矛盾

**表现**：`README.md:71-78` 说"Installation instructions will be added once packaging format is finalized"；`USAGE.md:24-28` 给出完整安装流程。新用户从 GitHub 落地找不到入口。

#### 2.1 🔴 stop.sh 22 处静默失败但 0 处 user-facing 日志

**表现**：`stop.sh:4` 定义了 `die()` 但**全文 0 处调用**。`multica issue comment add 2>/dev/null || true`（line 111、123、177、183）、`git commit ... 2>/dev/null || true`（line 123）、`sed -i ... 2>/dev/null || true`（line 169）——全部静默。用户根本不知道是 stop hook 失败了。

**建议**：(a) 失败追加到 `.multica/state/hook-errors.log`（永不阻断 hook，但保留证据）；(b) session-start.sh 启动时若 hook-errors.log non-empty，注入 warning 到 context；(c) 提供 `multica-plugin doctor` 汇总最近错误。

#### 2.2 🟠 pre-tool.sh fail-closed 但 user feedback 在 issue 评论

**表现**：`pre-tool.sh:42-50` 在 `multica safe-exec` 不可用时 exit 1 阻塞工具调用。当无 `MULTICA_ISSUE_ID` 时（line 21-26），post_comment 静默跳过——本地手动启动 Claude Code 测试时，工具被阻塞却没有任何错误提示。

**建议**：无 issue 上下文且 safe-exec 缺失时 fail-open 并日志记录；或打印明确 stderr：`[pre-tool] BLOCKED: multica safe-exec unavailable. To bypass: unset MULTICA_PLUGIN_ROOT.`

#### 2.3 🟡 session-start.sh JSON 解析回退到 awk 时寂静

**表现**：`hooks/session-start.sh:100-114` 当 python3 缺失时回退到脆弱的 awk 解析，但无任何提示说"降级到 awk 解析"。

**建议**：回退分支注入 context warning："Note: python3 unavailable, prior-learnings parse may be incomplete."

#### 3.1 🟠 12+ 概念无分层、无学习路径

**建议**：(a) 提供"5 分钟上手"：装 plugin → 跑 smoke test → 看一条假 issue 流转；(b) 把 Squad / Persistence Loop / Subagent dispatch 移到 advanced.md，default workflow 只讲 5-phase + HITL + Notepad。

#### 3.2 🟠 术语爆炸且未集中定义

**表现**：`<promise>DONE</promise>`、`[HITL]`、`[HITL:leader]`、`[HITL:human]`、`[checkpoint:hash]`、`[verification]`、`[loop-complete]`、`[capability=missing:destructive-guard]`、`squad-activity.marker`、`squad-audit-warning`、`hitl-bounces.json`、`consolidation-prompt.txt`、`loop.json`、`current_issue`——分散在多文件中。

## 六、Open Questions（需要产品决策）

1. **插件的首要用户**到底是谁？plugin operator、issue reviewer、还是 agent？三者冲突时谁优先？— 决定所有后续设计取舍。
2. **Multica daemon reaper 对 blocked HITL 的 SLA** 是什么？多少小时不回复会触发什么？— 决定 HITL 协议是否需要补 nudge 机制。
3. **5-phase / 3-strike / 35%/25% context 阈值** 有数据支撑还是工程拍定？— 决定是否需要可配置或重新校准。
4. **v0.4.0 learnings.jsonl + git commit 同步路径** 是 self-host 假设吗？SaaS 版本如何？— 决定知识管理架构方向。
5. **consolidation-prompt.txt 由谁消费、何时清理**？当前是死代码还是待接入功能？
6. **`multica issue rerun <id>`** 是否真实存在于 multica CLI？— 直接影响 USAGE.md §5.2 恢复路径有效性。
7. **settings.json 内的 hooks 路径**是安装时固化的绝对路径还是运行时展开的占位符？— 直接影响 MULTICA_PLUGIN_ROOT env var 是否真正必要。
8. **HITL 收到非 A/B 选项的 reviewer 回复时，agent 怎么处理**？— 决定是否要扩展 HITL 状态机。
